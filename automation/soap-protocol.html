1. What Is SOAP?

SOAP (Simple Object Access Protocol) is a messaging protocol specification for exchanging structured information in the implementation of web services. It uses XML as its message format and relies on application layer protocols — most commonly HTTP or SMTP — for message negotiation and transmission.

Key Characteristics

  • Protocol-based: SOAP is a strict protocol with a well-defined specification (W3C standard), not merely an architectural style.
  • XML-only: All SOAP messages are encoded in XML, providing platform and language independence.
  • Transport-agnostic: While HTTP is the most common transport, SOAP can work over SMTP, TCP, JMS, or any other protocol.
  • Contract-driven: Services are described using WSDL (Web Services Description Language) files that act as a formal contract between provider and consumer.
  • Built-in error handling: The SOAP Fault element provides a standardized mechanism for error reporting.
  • Extensible: WS-* standards (WS-Security, WS-ReliableMessaging, WS-AtomicTransaction, etc.) extend SOAP's capabilities for enterprise use cases.

Core Standards

Standard Purpose Maintained By
SOAP 1.1 / 1.2 Message envelope and encoding rules W3C
WSDL 1.1 / 2.0 Service interface description W3C
UDDI Service registry and discovery OASIS
WS-Security Message-level security (encryption, signing) OASIS
WS-ReliableMessaging Guaranteed message delivery OASIS
WS-AtomicTransaction Distributed transaction support OASIS

2. History & Evolution

Year Milestone
1998 XML-RPC created by Dave Winer (UserLand) and Microsoft — the precursor to SOAP.
1999 SOAP 1.0 developed by Microsoft, DevelopMentor, and UserLand Software.
2000 SOAP 1.1 submitted to W3C as a W3C Note.
2003 SOAP 1.2 becomes a W3C Recommendation. The acronym is dropped — "SOAP" is now just a name.
2004–2010 WS-* standards mature (WS-Security, WS-Policy, WS-Addressing, etc.).
2010+ REST gains dominance for new APIs; SOAP remains prevalent in enterprise, financial, and government systems.

3. How SOAP Works

Communication Flow

SOAP follows a request-response model (though one-way messaging is also supported). Here is the typical flow:

Step-by-Step Explanation

  1. Service Discovery: The client retrieves the WSDL file (usually via a ?wsdl URL) to learn about available operations, message formats, data types, and endpoint addresses.
  2. Request Construction: The client constructs a SOAP XML message with the proper envelope, headers, and body containing the operation name and parameters.
  3. Transport: The message is sent via HTTP POST (most common). The Content-Type header is set to text/xml (SOAP 1.1) or application/soap+xml (SOAP 1.2). A SOAPAction HTTP header identifies the operation.
  4. Server Processing: The server's SOAP engine deserializes the XML, routes to the appropriate service implementation, and executes the business logic.
  5. Response: The server packages the result (or fault) into a SOAP response envelope and sends it back.
  6. Client Processing: The client deserializes the response XML and processes the result.

Transport Bindings

Transport Use Case Content-Type
HTTP/HTTPS Most common; synchronous request-response text/xml (1.1) / application/soap+xml (1.2)
SMTP Asynchronous messaging via email text/xml
JMS Java Message Service queues — enterprise messaging Application-defined
TCP Raw socket communication for performance N/A

4. SOAP Message Structure

Every SOAP message is an XML document with a strict envelope structure:

4.1 SOAP Envelope

The root element that identifies the XML document as a SOAP message.

<?xml version="1.0" encoding="UTF-8"?>
<soap:Envelope
    xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
    xmlns:ws="http://example.com/webservice">

    <soap:Header>
        <!-- Optional headers -->
    </soap:Header>

    <soap:Body>
        <!-- Request or Response payload -->
    </soap:Body>

</soap:Envelope>

4.2 SOAP Header (Optional)

Contains metadata for processing — authentication tokens, transaction IDs, routing information, etc.

<soap:Header>
    <wsse:Security xmlns:wsse="http://docs.oasis-open.org/wss/2004/01/oasis-200401-wss-wssecurity-secext-1.0.xsd">
        <wsse:UsernameToken>
            <wsse:Username>admin</wsse:Username>
            <wsse:Password Type="PasswordText">secret</wsse:Password>
        </wsse:UsernameToken>
    </wsse:Security>
</soap:Header>

4.3 SOAP Body

Contains the actual request or response data — the operation being called and its parameters.

Request Example

<soap:Body>
    <ws:GetUserDetails>
        <ws:UserId>12345</ws:UserId>
    </ws:GetUserDetails>
</soap:Body>

Response Example

<soap:Body>
    <ws:GetUserDetailsResponse>
        <ws:User>
            <ws:Id>12345</ws:Id>
            <ws:Name>John Doe</ws:Name>
            <ws:Email>john.doe@example.com</ws:Email>
            <ws:Role>Admin</ws:Role>
        </ws:User>
    </ws:GetUserDetailsResponse>
</soap:Body>

4.4 SOAP Fault

A standardized error structure returned in the body when processing fails.

<soap:Body>
    <soap:Fault>
        <soap:Code>
            <soap:Value>soap:Sender</soap:Value>
        </soap:Code>
        <soap:Reason>
            <soap:Text xml:lang="en">User ID 99999 not found</soap:Text>
        </soap:Reason>
        <soap:Detail>
            <ws:ErrorCode>USER_NOT_FOUND</ws:ErrorCode>
            <ws:Timestamp>2026-03-31T10:30:00Z</ws:Timestamp>
        </soap:Detail>
    </soap:Fault>
</soap:Body>

SOAP 1.2 Fault Codes

Fault Code Meaning
soap:Sender The message was incorrectly formed or missing information (client error).
soap:Receiver The server could not process the message due to an internal error.
soap:VersionMismatch The SOAP version in the envelope namespace is not supported.
soap:MustUnderstand A mandatory header entry was not understood by the receiver.
soap:DataEncodingUnknown The data encoding style is not supported.

5. WSDL — Web Services Description Language

WSDL is an XML-based language that describes the public interface of a SOAP web service. It tells consumers what operations are available, what messages are exchanged, and where the service is located.

WSDL Structure

@startuml skinparam backgroundColor transparent skinparam defaultFontSize 13 skinparam rectangle { RoundCorner 10 } skinparam ArrowThickness 2 skinparam ArrowColor #475569 rectangle "**WSDL Document**" as WSDL #dbeafe;line:2563eb;line.bold rectangle "\nXSD Schema definitions" as types #f1f5f9;line:64748b rectangle "\nAbstract message parts" as message #f1f5f9;line:64748b rectangle "\nOperations (interface)" as portType #f1f5f9;line:64748b rectangle "\nProtocol & encoding details" as binding #f1f5f9;line:64748b rectangle "\nEndpoint URL(s)" as service #ecfdf5;line:10b981;line.bold WSDL --> types types --> message message --> portType portType --> binding binding --> service @enduml
Element Description
<types> Defines data types using XML Schema (XSD). Specifies the structure of request/response messages.
<message> Defines abstract messages composed of parts, each referencing an XSD type.
<portType> Defines the abstract set of operations (like a Java interface). Each operation maps input/output messages.
<binding> Specifies the protocol (SOAP) and encoding details for each operation.
<service> Defines the endpoint URL(s) where the service can be accessed.

WSDL Example

<?xml version="1.0" encoding="UTF-8"?>
<definitions name="UserService"
    xmlns="http://schemas.xmlsoap.org/wsdl/"
    xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/"
    xmlns:tns="http://example.com/userservice"
    xmlns:xsd="http://www.w3.org/2001/XMLSchema"
    targetNamespace="http://example.com/userservice">

    <!-- Data Types -->
    <types>
        <xsd:schema targetNamespace="http://example.com/userservice">
            <xsd:element name="GetUserRequest">
                <xsd:complexType>
                    <xsd:sequence>
                        <xsd:element name="UserId" type="xsd:int"/>
                    </xsd:sequence>
                </xsd:complexType>
            </xsd:element>
            <xsd:element name="GetUserResponse">
                <xsd:complexType>
                    <xsd:sequence>
                        <xsd:element name="Name" type="xsd:string"/>
                        <xsd:element name="Email" type="xsd:string"/>
                        <xsd:element name="Role" type="xsd:string"/>
                    </xsd:sequence>
                </xsd:complexType>
            </xsd:element>
        </xsd:schema>
    </types>

    <!-- Messages -->
    <message name="GetUserInput">
        <part name="parameters" element="tns:GetUserRequest"/>
    </message>
    <message name="GetUserOutput">
        <part name="parameters" element="tns:GetUserResponse"/>
    </message>

    <!-- Port Type (Interface) -->
    <portType name="UserServicePortType">
        <operation name="GetUser">
            <input message="tns:GetUserInput"/>
            <output message="tns:GetUserOutput"/>
        </operation>
    </portType>

    <!-- Binding -->
    <binding name="UserServiceBinding" type="tns:UserServicePortType">
        <soap:binding style="document"
            transport="http://schemas.xmlsoap.org/soap/http"/>
        <operation name="GetUser">
            <soap:operation soapAction="http://example.com/userservice/GetUser"/>
            <input><soap:body use="literal"/></input>
            <output><soap:body use="literal"/></output>
        </operation>
    </binding>

    <!-- Service Endpoint -->
    <service name="UserService">
        <port name="UserServicePort" binding="tns:UserServiceBinding">
            <soap:address location="https://api.example.com/UserService"/>
        </port>
    </service>

</definitions>

6. SOAP vs REST

Aspect SOAP REST
Type Protocol Architectural style
Data Format XML only JSON, XML, HTML, plain text, etc.
Transport HTTP, SMTP, JMS, TCP Primarily HTTP/HTTPS
Contract WSDL (formal, machine-readable) OpenAPI/Swagger (optional)
Statefulness Can be stateful or stateless Stateless by design
Security WS-Security (message-level) TLS/SSL (transport-level)
Transactions WS-AtomicTransaction (ACID) No built-in support
Reliability WS-ReliableMessaging Application-managed retries
Error Handling SOAP Fault (standardized) HTTP status codes
Performance Higher overhead (XML parsing) Lightweight (especially JSON)
Caching Not natively supported Built-in HTTP caching
Tooling Strong (auto-generated clients from WSDL) Simple (curl, Postman, any HTTP client)

7. When to Use SOAP

SOAP Is a Good Fit When...

  • Enterprise-grade security is required (WS-Security with XML signatures and encryption)
  • ACID-compliant distributed transactions are needed
  • Formal contracts between systems are mandatory (regulated industries)
  • Guaranteed message delivery is critical (WS-ReliableMessaging)
  • Integrating with legacy enterprise systems (ERP, banking, insurance)
  • Asynchronous processing via message queues (JMS)

REST Is Preferred When...

  • Building public-facing APIs with broad client support
  • Lightweight, fast, bandwidth-efficient communication is needed
  • Mobile or browser-based clients are the primary consumers
  • CRUD operations map cleanly to HTTP methods
  • Caching is important for performance
  • Rapid development and simplicity are priorities

Real-World Usage: SOAP remains heavily used in financial services (payment gateways like PayPal's classic API), healthcare (HL7/CDA), telecommunications, government systems, and enterprise integration platforms (SAP, Salesforce, Guidewire).

10. Testing with SoapUI (Java)

SoapUI is the most popular tool for SOAP testing. It's a Java-based application that can import WSDL files and auto-generate requests for every operation.

10.1 Getting Started

  1. Download SoapUI from https://www.soapui.org (open-source edition).
  2. Create a new SOAP project: File → New SOAP Project.
  3. Enter the WSDL URL (e.g., https://api.example.com/UserService?wsdl).
  4. SoapUI parses the WSDL and generates sample requests for each operation.

10.2 Creating Test Suites

  1. Right-click the project → New TestSuite.
  2. Add a TestCase → Add SOAP Request test steps.
  3. Add Assertions to validate responses:
    • Contains: Check for specific text in the response.
    • XPath Match: Assert specific XML element values.
    • Schema Compliance: Validate against the WSDL/XSD schema.
    • SOAP Fault: Assert that a fault IS or IS NOT returned.
    • Response SLA: Assert response time is under a threshold.

10.3 Groovy Script Assertions (JVM-based)

// Groovy script assertion in SoapUI
import groovy.xml.XmlSlurper

def response = new XmlSlurper().parseText(messageExchange.responseContent)

// Assert user name
def userName = response.Body.GetUserDetailsResponse.User.Name.text()
assert userName == "John Doe" : "Expected 'John Doe' but got '${userName}'"

// Assert email format
def email = response.Body.GetUserDetailsResponse.User.Email.text()
assert email.matches(/^[\w.]+@[\w.]+\.\w+$/) : "Invalid email format: ${email}"

// Assert response time
def responseTime = messageExchange.timeTaken
assert responseTime < 2000 : "Response too slow: ${responseTime}ms"

log.info "All assertions passed!"

10.4 Data-Driven Testing

SoapUI supports data-driven tests using DataSource test steps:

  1. Add a DataSource step (Excel, CSV, JDBC, or Groovy).
  2. Reference variables in your SOAP request: ${DataSource#UserId}.
  3. Add a DataSource Loop step to iterate through all rows.
  4. Each iteration sends a request with different data and validates the response.

11.2 Using JAX-WS Generated Client

import com.example.userservice.client.*;
import org.junit.jupiter.api.BeforeAll;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class UserServiceTest {

    private static UserServicePortType port;

    @BeforeAll
    static void setup() {
        UserService service = new UserService();
        port = service.getUserServicePort();
    }

    @Test
    void shouldReturnUserDetails() {
        GetUserRequest request = new GetUserRequest();
        request.setUserId(12345);

        GetUserResponse response = port.getUser(request);

        assertNotNull(response);
        assertEquals("John Doe", response.getName());
        assertEquals("john.doe@example.com", response.getEmail());
        assertEquals("Admin", response.getRole());
    }

    @Test
    void shouldReturnFaultForInvalidUser() {
        GetUserRequest request = new GetUserRequest();
        request.setUserId(-1);

        assertThrows(SOAPFaultException.class, () -> {
            port.getUser(request);
        });
    }
}

11.3 Raw SOAP Request with HttpURLConnection

import java.io.*;
import java.net.HttpURLConnection;
import java.net.URI;
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class RawSoapRequestTest {

    private static final String ENDPOINT = "https://api.example.com/UserService";

    @Test
    void shouldGetUserViaRawSoapRequest() throws Exception {
        String soapRequest = """
            <?xml version="1.0" encoding="UTF-8"?>
            <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
                           xmlns:ws="http://example.com/userservice">
                <soap:Body>
                    <ws:GetUserDetails>
                        <ws:UserId>12345</ws:UserId>
                    </ws:GetUserDetails>
                </soap:Body>
            </soap:Envelope>
            """;

        HttpURLConnection conn = (HttpURLConnection) URI.create(ENDPOINT).toURL().openConnection();
        conn.setRequestMethod("POST");
        conn.setRequestProperty("Content-Type", "application/soap+xml; charset=utf-8");
        conn.setDoOutput(true);

        try (OutputStream os = conn.getOutputStream()) {
            os.write(soapRequest.getBytes("UTF-8"));
        }

        assertEquals(200, conn.getResponseCode());

        String response;
        try (BufferedReader br = new BufferedReader(
                new InputStreamReader(conn.getInputStream(), "UTF-8"))) {
            StringBuilder sb = new StringBuilder();
            String line;
            while ((line = br.readLine()) != null) {
                sb.append(line);
            }
            response = sb.toString();
        }

        assertTrue(response.contains("John Doe"));
        assertTrue(response.contains("GetUserDetailsResponse"));
    }
}

11.4 XML Validation with XMLUnit

import org.junit.jupiter.api.Test;
import org.xmlunit.builder.DiffBuilder;
import org.xmlunit.diff.Diff;
import org.xmlunit.matchers.HasXPathMatcher;
import org.xmlunit.validation.Languages;
import org.xmlunit.validation.ValidationResult;
import org.xmlunit.validation.Validator;

import javax.xml.transform.stream.StreamSource;
import java.io.StringReader;

import static org.hamcrest.MatcherAssert.assertThat;
import static org.junit.jupiter.api.Assertions.*;
import static org.xmlunit.matchers.HasXPathMatcher.hasXPath;

class XmlValidationTest {

    @Test
    void shouldValidateResponseAgainstSchema() {
        String xsd = "<!-- XSD schema content -->";
        String responseXml = "<!-- SOAP response body XML -->";

        Validator validator = Validator.forLanguage(Languages.W3C_XML_SCHEMA_NS_URI);
        validator.setSchemaSource(new StreamSource(new StringReader(xsd)));

        ValidationResult result = validator.validateInstance(
            new StreamSource(new StringReader(responseXml)));

        assertTrue(result.isValid(), "Response does not match XSD schema");
    }

    @Test
    void shouldMatchExpectedXmlStructure() {
        String expected = "<User><Name>John Doe</Name></User>";
        String actual = "<User><Name>John Doe</Name></User>";

        Diff diff = DiffBuilder.compare(expected)
            .withTest(actual)
            .ignoreWhitespace()
            .build();

        assertFalse(diff.hasDifferences(), diff.toString());
    }

    @Test
    void shouldAssertXPathValues() {
        String xml = """
            <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
                <soap:Body>
                    <GetUserResponse xmlns="http://example.com/userservice">
                        <Name>John Doe</Name>
                        <Email>john@example.com</Email>
                    </GetUserResponse>
                </soap:Body>
            </soap:Envelope>
            """;

        assertThat(xml, hasXPath("//Name"));
        assertThat(xml, hasXPath("//Email"));
    }
}

REST Assured, despite its name, can send any HTTP request — including SOAP XML payloads. It provides a fluent Java API for constructing requests and asserting responses.

SOAP Test with REST Assured

import io.restassured.RestAssured;
import io.restassured.response.Response;
import org.junit.jupiter.api.Test;

import static io.restassured.RestAssured.given;
import static org.hamcrest.Matchers.*;

class SoapRestAssuredTest {

    private static final String ENDPOINT = "https://api.example.com/UserService";

    @Test
    void shouldGetUserDetailsViaSoap() {
        String requestBody = """
            <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
                           xmlns:ws="http://example.com/userservice">
                <soap:Body>
                    <ws:GetUserDetails>
                        <ws:UserId>12345</ws:UserId>
                    </ws:GetUserDetails>
                </soap:Body>
            </soap:Envelope>
            """;

        given()
            .contentType("application/soap+xml; charset=utf-8")
            .body(requestBody)
        .when()
            .post(ENDPOINT)
        .then()
            .statusCode(200)
            .body(hasXPath("//GetUserDetailsResponse/User/Name",
                equalTo("John Doe")))
            .body(hasXPath("//GetUserDetailsResponse/User/Email",
                equalTo("john.doe@example.com")))
            .time(lessThan(3000L));
    }

    @Test
    void shouldReturnSoapFaultForMissingUserId() {
        String requestBody = """
            <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
                           xmlns:ws="http://example.com/userservice">
                <soap:Body>
                    <ws:GetUserDetails>
                        <!-- UserId intentionally omitted -->
                    </ws:GetUserDetails>
                </soap:Body>
            </soap:Envelope>
            """;

        given()
            .contentType("application/soap+xml; charset=utf-8")
            .body(requestBody)
        .when()
            .post(ENDPOINT)
        .then()
            .statusCode(500)
            .body(hasXPath("//Fault/Code/Value",
                containsString("Sender")));
    }

    @Test

15. Best Practices

Testing Best Practices

  1. Always validate against the WSDL/XSD schema — don't just check individual fields. Schema validation catches structural regressions early.
  2. Use XPath for assertions, not string matching — XML formatting can change (whitespace, prefixes) without affecting semantics.
  3. Test all SOAP operations — auto-generate test stubs from the WSDL using tools like JAX-WS wsimport or SoapUI.
  4. Test SOAP Faults explicitly — verify fault codes, fault strings, and detail elements match expectations.
  5. Mock external dependencies — use WireMock or SoapUI MockService to isolate the service under test.
  6. Parameterize test data — use data-driven approaches (CSV, Excel, DataSource) rather than duplicating test cases.
  7. Validate namespaces — SOAP is namespace-sensitive. Tests should verify the correct namespace URIs.
  8. Test WS-Security independently — token validation, signature verification, and encryption should each have dedicated tests.
  9. Measure response times — add SLA assertions to catch performance regressions.
  10. Version your WSDL — track changes to the service contract and run regression tests when the WSDL changes.

Development Best Practices

  1. Contract-first development: Define the WSDL/XSD before writing code. This ensures a clean contract.
  2. Use document/literal style over RPC/encoded — it's the WS-I Basic Profile recommendation.
  3. Keep messages small: XML verbosity is SOAP's biggest performance cost. Avoid unnecessarily deep structures.
  4. Use MTOM for binary data: Don't Base64-encode large binaries in XML — use MTOM (Message Transmission Optimization Mechanism).
  5. Implement proper error handling: Return meaningful SOAP Faults with application-specific detail elements.

16. Common SOAP Faults & Troubleshooting

Issue Symptom Solution
Namespace mismatch Server returns "unexpected element" or ignores request body Compare request namespaces against WSDL. Ensure target namespace matches exactly.
SOAPAction header missing HTTP 500 or "operation not found" Add the correct SOAPAction HTTP header as defined in the WSDL binding.
Wrong Content-Type Server rejects request or parses incorrectly Use text/xml for SOAP 1.1, application/soap+xml for SOAP 1.2.
WSDL unreachable Client cannot generate stubs Verify network connectivity. Try accessing ?wsdl URL directly in a browser.
SSL/TLS errors Connection refused or certificate errors Import the server's certificate into the Java truststore: keytool -import.
XML parsing failure "Content is not allowed in prolog" error Remove BOM characters. Ensure XML declaration encoding matches actual encoding.
Timeout errors Connection or read timeout Configure client timeouts. Check server health. Increase timeout for long operations.
WS-Security rejection "Security header invalid" fault Verify timestamp freshness, username/password, and that the security policy matches.

Debugging Tip: Enable SOAP message logging in Java using system properties:
-Dcom.sun.xml.ws.transport.http.client.HttpTransportPipe.dump=true
This prints the full SOAP request/response XML to stdout, which is invaluable for troubleshooting.

17. Security Testing for SOAP

SOAP services are vulnerable to specific attack vectors due to their XML-based nature. Testing should cover the following:

17.1 Common Vulnerabilities

Vulnerability Description Test Approach
XML External Entity (XXE) Attacker injects external entity references to read files or cause SSRF Send requests with <!DOCTYPE> declarations containing external entities. Verify they are rejected.
XML Bomb (Billion Laughs) Recursive entity expansion causes DoS via memory exhaustion Send deeply nested entity definitions. Verify server limits entity expansion.
SOAP Injection Injecting XML into user-controlled fields to alter SOAP message structure Include XML special characters (< > &) in input fields. Verify proper encoding.
WS-Security Bypass Removing or modifying security headers to bypass authentication Send requests without wsse:Security headers. Verify rejection.
WSDL Exposure Public WSDL reveals internal service structure and operations Verify that production WSDL is not publicly accessible (or is appropriately limited).
XPath Injection Manipulating XPath queries through user input Include XPath syntax (' or 1=1) in input fields. Verify server does not evaluate them.

17.2 XXE Test Example (Java)

@Test
void shouldRejectXxePayload() {
    // This payload attempts to read a server file via XXE
    String xxePayload = """
        <?xml version="1.0" encoding="UTF-8"?>
        <!DOCTYPE foo [
            <!ENTITY xxe SYSTEM "file:///etc/passwd">
        ]>
        <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"
                       xmlns:ws="http://example.com/userservice">
            <soap:Body>
                <ws:GetUserDetails>
                    <ws:UserId>&xxe;</ws:UserId>
                </ws:GetUserDetails>
            </soap:Body>
        </soap:Envelope>
        """;

    given()
        .contentType("application/soap+xml; charset=utf-8")
        .body(xxePayload)
    .when()
        .post(ENDPOINT)
    .then()
        // Server should reject DTD/external entities
        .statusCode(anyOf(is(400), is(500)))
        .body(not(containsString("root:")));  // Ensure file content NOT returned
}

@Test
