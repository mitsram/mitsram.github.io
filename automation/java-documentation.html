1. Overview & History

Java is a general-purpose, class-based, object-oriented programming language designed to have as few implementation dependencies as possible. It follows the principle of "Write Once, Run Anywhere" (WORA), meaning compiled Java code can run on all platforms that support Java without the need for recompilation.

2. Java Platform Architecture

The Java platform consists of several interrelated components that together provide a complete environment for developing and running applications.

┌─────────────────────────────────────────────────────────────────┐ │ Java Application │ ├─────────────────────────────────────────────────────────────────┤ │ Java Standard Library │ │ (java.lang, java.util, java.io, java.net, ...) │ ├─────────────────────────────────────────────────────────────────┤ │ Java Development Kit (JDK) │ │ ┌──────────────┐ ┌──────────────┐ ┌──────────────────┐ │ │ │ javac │ │ javadoc │ │ jshell │ │ │ │ (compiler) │ │ (docs) │ │ (REPL) │ │ │ └──────────────┘ └──────────────┘ └──────────────────┘ │ ├─────────────────────────────────────────────────────────────────┤ │ Java Runtime Environment (JRE) │ │ ┌─────────────────────────────────────────────────────────┐ │ │ │ Java Virtual Machine (JVM) │ │ │ │ ┌─────────────┐ ┌────────────┐ ┌──────────────────┐ │ │ │ │ │ Class │ │ Execution │ │ Garbage │ │ │ │ │ │ Loader │ │ Engine │ │ Collector │ │ │ │ │ └─────────────┘ └────────────┘ └──────────────────┘ │ │ │ │ ┌─────────────┐ ┌────────────┐ ┌──────────────────┐ │ │ │ │ │ JIT │ │ Runtime │ │ Native Method │ │ │ │ │ │ Compiler │ │ Data Areas │ │ Interface (JNI) │ │ │ │ │ └─────────────┘ └────────────┘ └──────────────────┘ │ │ │ └─────────────────────────────────────────────────────────┘ │ ├─────────────────────────────────────────────────────────────────┤ │ Operating System / Hardware │ └─────────────────────────────────────────────────────────────────┘

Platform Editions

Java SE (Standard Edition)

Core platform. Includes the JVM, standard libraries, and development tools. Foundation for all Java development.

Jakarta EE (Enterprise Edition)

Extends Java SE with specifications for enterprise features: servlets, JPA, CDI, JMS, JAX-RS, and more. Managed by Eclipse Foundation.

Java ME (Micro Edition)

Subset for embedded and mobile devices. Largely superseded by Android (which uses its own runtime) and IoT-specific solutions.

Key Components

JDK (Java Development Kit)

The complete development environment including the compiler (javac), debugger (jdb), archiver (jar), documentation generator (javadoc), and the JRE. In Java 21, the JRE is no longer distributed separately — the JDK is the single distribution.

JRE (Java Runtime Environment)

Contains the JVM, core libraries, and supporting files needed to run Java applications. Since Java 11, Oracle no longer ships a standalone JRE; the jlink tool allows creation of custom runtime images.

JVM (Java Virtual Machine)

An abstract computing machine that enables the computer to run Java bytecode. The JVM is platform-specific but provides a platform-independent execution environment. Major implementations include HotSpot (Oracle), OpenJ9 (Eclipse), and GraalVM.

3. How Java Works — Compilation & Execution

The Compilation Pipeline

┌────────────────┐ javac ┌────────────────┐ JVM ┌────────────────┐ │ Source Code │ ────────────▶ │ Bytecode │ ───────────▶ │ Machine Code │ │ (.java) │ Compile │ (.class) │ Execute │ (Native) │ └────────────────┘ Time └────────────────┘ Runtime └────────────────┘ │ ▼ ┌────────────────┐ │ JAR / Module │ │ Packaging │ └────────────────┘

Step-by-step Process

  1. Write — Developer writes source code in .java files.
  2. Compile — The javac compiler parses, validates, and transforms source code into platform-independent bytecode stored in .class files.
  3. Package — Class files are optionally bundled into .jar (Java Archive) or modular .jmod files.
  4. Load — The JVM's Class Loader subsystem locates and loads classes into memory on demand.
  5. Verify — The Bytecode Verifier ensures the bytecode is valid and does not violate security constraints.
  6. Interpret / Compile — The execution engine initially interprets bytecode. The JIT (Just-In-Time) compiler identifies "hot" code paths and compiles them to optimized native machine code.
  7. Execute — Native machine code runs on the underlying hardware.

Example: Hello World

// HelloWorld.java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, Java 21!");
    }
}
# Compile
$ javac HelloWorld.java

# Run
$ java HelloWorld
Hello, Java 21!

# Or use single-file source-code execution (Java 11+)
$ java HelloWorld.java

Java 21 Enhancement: Unnamed classes and instance main methods (JEP 463, preview) allow writing simpler programs without the public static void main boilerplate:

// In Java 21 (preview feature)
void main() {
    System.out.println("Hello, Java 21!");
}

4. JVM Deep Dive

Class Loader Subsystem

The class loader follows a delegation model with three layers:

Class Loader Responsibility
Bootstrap Class LoaderLoads core Java classes from java.base module (e.g., java.lang.Object, java.lang.String). Written in native code.
Platform Class LoaderLoads platform modules not in java.base (replaces the old Extension class loader).
Application Class LoaderLoads classes from the application classpath and module path.

Runtime Data Areas

┌──────────────────────────────────────────────────────┐ │ JVM Memory │ │ │ │ ┌─────────────────────┐ ┌────────────────────┐ │ │ │ Heap │ │ Method Area │ │ │ │ ┌───────────────┐ │ │ (Class metadata, │ │ │ │ │ Young Gen │ │ │ constant pool, │ │ │ │ │ ┌──────────┐ │ │ │ static vars) │ │ │ │ │ │ Eden │ │ │ └────────────────────┘ │ │ │ │ ├──────────┤ │ │ │ │ │ │ │ Survivor │ │ │ ┌────────────────────┐ │ │ │ │ └──────────┘ │ │ │ Stack (per thread) │ │ │ │ ├───────────────┤ │ │ │ ┌──────────────┐ │ │ │ │ │ Old Gen │ │ │ │ │ Local vars │ │ │ │ │ └───────────────┘ │ │ │ │ Operand stack│ │ │ │ └─────────────────────┘ │ │ │ Frame data │ │ │ │ │ │ └──────────────┘ │ │ │ ┌─────────────────────┐ │ └────────────────────┘ │ │ │ PC Register │ │ │ │ │ (per thread) │ │ ┌────────────────────┐ │ │ └─────────────────────┘ │ │ Native Method │ │ │ │ │ Stack │ │ │ │ └────────────────────┘ │ └──────────────────────────────────────────────────────┘
Area Description Thread Scope
HeapStores all object instances and arrays. Divided into Young and Old generations for GC purposes.Shared
Method AreaStores class structures, method data, constant pool, and static variables. Implemented as Metaspace since Java 8.Shared

Execution Engine

  • Interpreter — Reads and executes bytecode instructions one by one. Fast startup but slower sustained execution.
  • JIT Compiler (C1 & C2) — Compiles hot bytecode to optimized native code at runtime.
    • C1 (Client compiler) — Fast compilation with basic optimizations. Used for quick startup.
    • C2 (Server compiler) — Aggressive optimizations (inlining, escape analysis, loop unrolling). Used for long-running applications.
    • Tiered Compilation — Default in Java 21. Starts with C1, promotes hot methods to C2.
  • GraalVM JIT — Alternative high-performance JIT compiler written in Java, offering superior peak performance for certain workloads.

5. Language Fundamentals

Type System

Primitive Types

byte (8-bit), short (16-bit), int (32-bit), long (64-bit), float (32-bit FP), double (64-bit FP), boolean, char (16-bit Unicode).

Reference Types

Classes, interfaces, arrays, enums, records, annotations. All inherit from java.lang.Object. Stored on the heap.

Object-Oriented Principles

PolymorphismMethod overriding (runtime), method overloading (compile-time), interface default methods
AbstractionAbstract classes, interfaces, sealed classes/interfaces (Java 17+)

Key Language Constructs (Modern Java)

Records (Java 16+)

// Immutable data carrier — generates equals, hashCode, toString, accessors

var p = new Point(3, 4);
System.out.println(p.x()); // 3

Sealed Classes (Java 17+)

    permits Circle, Rectangle, Triangle {
    double area();

public record Rectangle(double w, double h) implements Shape {
    public double area() { return w * h; }

public record Triangle(double base, double height) implements Shape {
    public double area() { return 0.5 * base * height; }

6. Java 21 Features

Java 21 is a Long-Term Support (LTS) release, making it an excellent target for production systems. It includes 15 JEPs (JDK Enhancement Proposals), finalizing several features that were in preview in earlier releases.

Finalized Features

Virtual Threads (JEP 444)

Lightweight threads managed by the JVM rather than the OS. Enables highly concurrent applications with a simple thread-per-request model.

// Create and start a virtual thread
Thread.startVirtualThread(() -> {
    System.out.println("Running in virtual thread: " + Thread.currentThread());
});

// Using ExecutorService
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 10_000).forEach(i ->
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        })
    );
}

Pattern Matching for switch (JEP 441)

static String describe(Object obj) {
    return switch (obj) {
        case Integer i when i > 0  -> "Positive integer: " + i;
        case Integer i             -> "Non-positive integer: " + i;
        case String s              -> "String of length " + s.length();
        case Point(int x, int y)   -> "Point at (%d, %d)".formatted(x, y);
        case null                  -> "null value";
        default                    -> "Unknown: " + obj;
    };
}

Record Patterns (JEP 440)

// Deconstruct records in pattern matching
record Pair<T>(T first, T second) { }

static void printPair(Object obj) {
    if (obj instanceof Pair(String a, String b)) {
        System.out.println("Pair of strings: " + a + ", " + b);
    }
}

Sequenced Collections (JEP 431)

// New interfaces: SequencedCollection, SequencedSet, SequencedMap
SequencedCollection<String> list = new ArrayList<>(List.of("a", "b", "c"));

list.getFirst();      // "a"
list.getLast();        // "c"
list.reversed();      // reversed view: ["c", "b", "a"]

SequencedMap<String, Integer> map = new LinkedHashMap<>();
map.put("one", 1);
map.put("two", 2);
map.firstEntry();     // one=1
map.lastEntry();      // two=2

String Templates (JEP 430 — Preview)

// Preview feature — use --enable-preview flag
String name = "Java";
int version = 21;
String message = STR."Welcome to \{name} \{version}!";

7. Memory Model & Garbage Collection

Java Memory Model (JMM)

The JMM defines how threads interact through memory and what behaviors are allowed in concurrent execution. Key concepts:

  • Happens-before relationship — Guarantees visibility of writes across threads (e.g., unlocking a monitor happens-before subsequent locking).
  • volatile — Ensures visibility and ordering of a variable across threads without locking.
  • final fields — Guaranteed to be visible to all threads after construction completes.

Garbage Collectors in Java 21

Collector Description Best For
G1 GC (default)Region-based collector balancing throughput and latency. Divides heap into regions, collects garbage-first regions.General-purpose workloads
ZGC (Generational)Ultra-low latency collector. Pause times < 1ms regardless of heap size. Java 21 adds generational support (JEP 439).Low-latency applications, large heaps (multi-TB)
ShenandoahLow-pause collector that performs concurrent compaction. Available in OpenJDK distributions.Low-latency requirements
Parallel GCThroughput-oriented collector using multiple threads for young and old generation collection.Batch processing, throughput priority
Serial GCSingle-threaded collector with minimal footprint.Small heaps, single-core or containerized environments
Epsilon GCNo-op collector — allocates memory but never reclaims it.Performance testing, short-lived processes

Common GC Tuning Flags

# Use Generational ZGC
java -XX:+UseZGC -XX:+ZGenerational -Xmx4g MyApp

# Use G1 with tuning
java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -Xms2g -Xmx4g MyApp

# GC logging (Java 21)
java -Xlog:gc*:file=gc.log:time,uptime,level,tags MyApp

8. Concurrency & Virtual Threads

Traditional Threading Model

Java threads map 1:1 to OS threads (platform threads). Creating thousands is expensive due to memory overhead (~1MB stack per thread) and OS scheduling costs.

Virtual Threads (Project Loom)

Virtual threads are lightweight threads managed by the JVM. They are mounted on platform threads (carrier threads) and can be unmounted when performing blocking operations (I/O, sleep, locks).

┌────────────────────────────────────────────────────────────┐ │ JVM Scheduler │ │ │ │ Virtual Thread 1 ──┐ │ │ Virtual Thread 2 ──┤ ┌──────────────┐ │ │ Virtual Thread 3 ──┼───▶│ Platform │──▶ OS Thread 1 │ │ Virtual Thread 4 ──┤ │ Thread Pool │ │ │ Virtual Thread 5 ──┤ │ (Carrier │──▶ OS Thread 2 │ │ ... ─┤ │ Threads) │ │ │ Virtual Thread N ──┘ │ │──▶ OS Thread M │ │ └──────────────┘ │ │ │ │ N can be millions M ≈ CPU cores │ └────────────────────────────────────────────────────────────┘ 
// Structured Concurrency (Preview in Java 21, JEP 453)
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<String>  user  = scope.fork(() -> fetchUser(id));
    Subtask<Integer> order = scope.fork(() -> fetchOrder(id));

    scope.join();           // Wait for both
    scope.throwIfFailed();  // Propagate errors

    return new Response(user.get(), order.get());
}

// Scoped Values (Preview in Java 21, JEP 446)
private static final ScopedValue<String> USER = ScopedValue.newInstance();

ScopedValue.where(USER, "admin").run(() -> {
    System.out.println(USER.get());  // "admin"
    processRequest();                 // USER is accessible in callees
});

9. Module System (JPMS)

Introduced in Java 9, the Java Platform Module System (Project Jigsaw) provides strong encapsulation and reliable configuration at the package level.

// module-info.java
module com.example.myapp {
    requires java.net.http;            // depends on HTTP client module
    requires java.sql;                 // depends on JDBC module
    requires transitive com.example.lib; // transitive dependency

    exports com.example.myapp.api;     // expose packages to other modules
    opens com.example.myapp.model to   // allow reflection (e.g., for Jackson)
        com.fasterxml.jackson.databind;
}

Key Module Commands

# Compile with modules
javac -d out --module-source-path src $(find src -name "*.java")

# Run a modular application
java --module-path out -m com.example.myapp/com.example.myapp.Main

# Create a custom runtime image
jlink --module-path out --add-modules com.example.myapp --output myapp-runtime

# List modules in a JDK
java --list-modules

10. Build Tools & Dependency Management

Maven

The most widely used build tool in the Java ecosystem. Uses an XML-based POM (Project Object Model) for declarative build configuration.

<!-- pom.xml -->
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>myapp</artifactId>
    <version>1.0.0</version>

    <properties>
        <maven.compiler.source>21</maven.compiler.source>
        <maven.compiler.target>21</maven.compiler.target>
    </properties>

    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <version>3.2.0</version>
        </dependency>
    </dependencies>
</project>
# Common Maven commands
mvn clean install          # Clean, compile, test, package, install
mvn test                   # Run tests
mvn package                # Create JAR/WAR
mvn dependency:tree        # Show dependency tree
mvn versions:display-dependency-updates  # Check for updates

Gradle

A flexible build tool using Groovy or Kotlin DSL. Faster than Maven due to incremental builds, build cache, and daemon process.

// build.gradle.kts (Kotlin DSL)
plugins {
    java
    id("org.springframework.boot") version "3.2.0"
}

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)
    }
}

repositories {
    mavenCentral()
}

dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.1")
}
# Common Gradle commands
./gradlew build            # Compile, test, package
./gradlew test             # Run tests
./gradlew dependencies     # Show dependency tree
./gradlew bootRun          # Run Spring Boot app

11. Testing Frameworks

JUnit 5 (Jupiter)

The de facto standard testing framework for Java. Modular architecture with JUnit Platform, JUnit Jupiter, and JUnit Vintage.

import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {

    private Calculator calc;

    @BeforeEach
    void setUp() {
        calc = new Calculator();
    }

    @Test
    @DisplayName("Addition of two positive numbers")
    void addPositiveNumbers() {
        assertEquals(5, calc.add(2, 3));
    }

    @ParameterizedTest
    @CsvSource({"1, 1, 2", "2, 3, 5", "10, -5, 5"})
    void addParameterized(int a, int b, int expected) {
        assertEquals(expected, calc.add(a, b));
    }

    @Test
    void divideByZeroThrowsException() {
        assertThrows(ArithmeticException.class,
            () -> calc.divide(1, 0));
    }
}

Mockito

The most popular mocking framework. Creates test doubles for unit testing in isolation.

import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository repository;

    @InjectMocks
    private OrderService service;

    @Test
    void findOrderReturnsCorrectOrder() {
        var order = new Order(1L, "Widget", 3);
        when(repository.findById(1L)).thenReturn(Optional.of(order));

        var result = service.findOrder(1L);

        assertEquals("Widget", result.product());
        verify(repository).findById(1L);
    }
}

15. Security

Built-in Security Features

  • Bytecode Verification — Class files are verified before execution, preventing invalid code.
  • Strong Typing — Eliminates entire categories of memory-related vulnerabilities (no pointer arithmetic, bounds checking).
  • Security Manager — (Deprecated in Java 17, removal pending) Sandboxing mechanism for restricting code access.
  • Cryptography — Java Cryptography Architecture (JCA/JCE) with pluggable providers.
  • TLS/SSL — Built-in support via JSSE. Java 21 supports TLS 1.3.
  • Key Encapsulation Mechanism (KEM) — New in Java 21 (JEP 452) for post-quantum key exchange preparation.

16. Best Practices

Code Style & Design

  • Prefer records for immutable data carriers over POJOs with boilerplate.
  • Use sealed classes to model closed type hierarchies.
  • Favor pattern matching (instanceof, switch) over manual type-checking and casting.
  • Prefer composition over inheritance. Use interfaces with default methods judiciously.
  • Return Optional<T> instead of null from methods that may not produce a result.
  • Use var for local variables when the type is obvious from context.
  • Follow package-by-feature over package-by-layer organization.

Performance

  • Use virtual threads for I/O-bound workloads instead of platform thread pools.
  • Profile before optimizing — use JFR + JMC or async-profiler.
  • Choose the right garbage collector for your workload (G1 for general, ZGC for low-latency).
  • Use StringBuilder for string concatenation in loops (though the compiler optimizes simple cases).
  • Prefer Streams for declarative collection processing; use parallel streams only for CPU-bound work on large datasets.
  • Use jlink to create minimal runtime images for production deployments.

Option 5: AWS Lambda (Serverless)

Deploy Java functions that scale automatically. Java 21 is supported as a managed runtime. Use SnapStart to reduce cold start times from ~5s to ~200ms.

Lambda Handler

// Using AWS Lambda Java Events library
public class OrderHandler implements RequestHandler<APIGatewayProxyRequestEvent,
                                                      APIGatewayProxyResponseEvent> {

    private final ObjectMapper mapper = new ObjectMapper();
    private final OrderService orderService = new OrderService();

    @Override
    public APIGatewayProxyResponseEvent handleRequest(
            APIGatewayProxyRequestEvent input, Context context) {

        try {
            var order = mapper.readValue(input.getBody(), OrderRequest.class);
            var result = orderService.createOrder(order);

            return new APIGatewayProxyResponseEvent()
                .withStatusCode(201)
                .withBody(mapper.writeValueAsString(result));

        } catch (Exception e) {
            context.getLogger().log("Error: " + e.getMessage());
            return new APIGatewayProxyResponseEvent()
                .withStatusCode(500)
                .withBody("{\"error\": \"Internal server error\"}");
        }
    }
}

// AWS CDK Stack in Java
public class MyAppStack extends Stack {

    public MyAppStack(final Construct scope, final String id) {
        super(scope, id);

        // VPC
        var vpc = Vpc.Builder.create(this, "MyAppVpc")
            .maxAzs(2)
            .build();

        // ECS Cluster

            .vpc(vpc)

        // Fargate Service with ALB

            .create(this, "MyAppService")
            .cluster(cluster)
            .cpu(1024)
            .memoryLimitMiB(2048)
            .desiredCount(2)
            .taskImageOptions(ApplicationLoadBalancedTaskImageOptions.builder()
                .image(ContainerImage.fromEcrRepository(
                    Repository.fromRepositoryName(this, "Repo", "myapp")))
                .containerPort(8080)
                .environment(Map.of(
                    "SPRING_PROFILES_ACTIVE", "production",
                    "JAVA_TOOL_OPTIONS", "-XX:+UseZGC -XX:+ZGenerational"
                ))
                .build())
            .publicLoadBalancer(true)
            .build();

        // Auto-scaling
        var scaling = service.getService().autoScaleTaskCount(
            EnableScalingProps.builder().minCapacity(2).maxCapacity(10).build());

        scaling.scaleOnCpuUtilization("CpuScaling",
            CpuUtilizationScalingProps.builder()
                .targetUtilizationPercent(70)
                .build());

        // RDS Database
        var db = DatabaseInstance.Builder.create(this, "MyAppDb")
            .engine(DatabaseInstanceEngine.postgres(
                PostgresInstanceEngineProps.builder()
                    .version(PostgresEngineVersion.VER_16).build()))
            .vpc(vpc)
            .instanceType(InstanceType.of(InstanceClass.T3, InstanceSize.MEDIUM))
            .credentials(Credentials.fromGeneratedSecret("dbadmin"))
            .build();

        db.getConnections().allowDefaultPortFrom(service.getService());
    }
}
