1. What is Service Virtualization?

Service Virtualization is the practice of simulating the behavior of dependent systems (APIs, third-party services, databases, message queues) so that the system under test can be developed and tested in isolation, without requiring the actual dependent services to be available.

In modern distributed architectures, applications depend on numerous external APIs, microservices, and third-party platforms. These dependencies introduce constraints: services may be unavailable, expensive to access, slow, or still under development. Service virtualization removes these constraints by providing realistic simulations that mimic the behavior of real services.

┌──────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ │ │ │ │ External API │ │ Application │───────▶ │ Virtual Service │ ──(X)──▶│ (Unavailable) │ │ Under Test │ │ (WireMock) │ │ │ │ │ │ │ └──────────────────┘ └──────────────┘ └──────────────────┘ │ ▼ Simulated Responses (Configured Stubs)

1.1 Why Service Virtualization?

Teams adopt service virtualization to solve real, tangible problems that slow down the development lifecycle:

Problem Without Service Virtualization With Service Virtualization
Dependency unavailability Teams are blocked, waiting for external services Simulate the service and keep working
Third-party rate limits / costs Pay per call or hit rate limits in testing Virtual services have no rate limits or costs
Data constraints Hard to reproduce edge cases with live data Full control over responses, errors, delays
Environment instability Flaky tests due to unreliable shared environments Deterministic, repeatable test environments
Parallel development Frontend waits for backend API to be ready Develop against virtual API contract
Fault testing Difficult to trigger real timeouts or 500 errors Simulate any failure scenario on demand

1.2 Service Virtualization vs. Mocking vs. Stubbing

These terms are often used interchangeably but have important distinctions:

Aspect Mocking Stubbing Service Virtualization
Scope In-process, code-level In-process, code-level Network-level (HTTP, TCP, JMS)
Protocol Language-specific (Java, Python) Language-specific Protocol-level (HTTP, HTTPS, SOAP)
Behavior Verifies interactions (was X called?) Returns canned responses Simulates realistic service behavior
Statefulness Typically stateless per test Stateless Can be stateful across requests
Shared access Same process only Same process only Multiple consumers over the network
Typical tools Mockito, unittest.mock Sinon.js, test doubles WireMock, Mountebank, Hoverfly

Key insight: Service virtualization operates at the network layer. The application under test makes real HTTP calls — it doesn't know the responses come from a virtual service. This makes virtual services usable by any technology stack.

1.3 Common Use Cases

  • Shift-left testing: Write integration tests before dependent services are built
  • Third-party API simulation: Payment gateways, SMS providers, identity providers
  • Edge case testing: Simulate 500 errors, timeouts, malformed responses, rate limiting
  • Performance testing: Control latency and throughput of virtualized dependencies
  • Contract validation: Verify that consumers and providers agree on API contracts
  • Demo environments: Run demos without relying on live infrastructure
  • Security testing: Simulate injection responses, malformed headers, unexpected payloads

2. What is WireMock?

WireMock is an open-source HTTP mock server for service virtualization. It creates a local or remote HTTP server that intercepts requests and returns configured responses, enabling you to simulate APIs for testing purposes. Originally built for the Java ecosystem, WireMock is now widely adopted across all technology stacks.

WireMock at a Glance:

  • HTTP/HTTPS stubbing based on request matching
  • Record traffic from real APIs and replay it
  • Simulate faults, delays, and network errors
  • Stateful scenarios for complex workflows
  • Response templating with dynamic data
  • Java API, REST API, JSON configuration, and Docker support
  • Available as standalone JAR, JUnit rule, or Docker image

2.1 Architecture & How It Works

WireMock runs as an HTTP server (by default on port 8080) that intercepts incoming requests and matches them against a set of configured stub mappings. When a match is found, the corresponding response is returned. When no match is found, a 404 is returned (or the request can be proxied to a real service).

┌─────────────────────────────────┐ │ WireMock Server │ │ │ ┌─────────────────┐ HTTP Request │ ┌───────────────────────────┐ │ │ Application / │ ──────────────▶ │ │ Request Matcher Engine │ │ │ Test Client │ │ │ │ │ │ │ ◀────────────── │ │ URL, Method, Headers, │ │ └─────────────────┘ HTTP Response │ │ Body, Query Params │ │ │ └─────────┬─────────────────┘ │ │ │ │ │ ▼ │ │ ┌───────────────────────────┐ │ │ │ Stub Mappings Store │ │ │ │ │ │ │ │ mapping-1.json │ │ │ │ mapping-2.json │ │ │ │ mapping-n.json │ │ │ └─────────┬─────────────────┘ │ │ │ │ │ ▼ │ │ ┌───────────────────────────┐ │ │ │ Response Definition │ │ │ │ │ │ │ │ Status, Headers, Body, │ │ │ │ Delays, Faults, │ │ │ │ Templated Responses │ │ │ └───────────────────────────┘ │ └─────────────────────────────────┘

Request Processing Flow

  1. Request received: WireMock receives an HTTP request from the application under test
  2. Matching: The request is evaluated against all configured stub mappings in priority order
  3. Response generation: If a match is found, the configured response is generated (optionally with templating)
  4. Response delivery: The response is sent back, with any configured delays or faults applied
  5. Journal logging: The request is recorded in the request journal for later verification

2.2 Key Features

Feature Description
Stubbing Define request-response pairs using JSON files or programmatic API
Request Matching Match by URL, method, headers, query parameters, cookies, and body content (JSON, XML, regex)
Response Templating Generate dynamic responses using Handlebars templates with access to request data
Record & Playback Proxy to a real API, record all interactions, then replay them without the real service
Stateful Scenarios Model sequences of interactions where responses change based on previous requests
Fault Injection Simulate connection resets, empty responses, random data, and malformed responses
Delay Simulation Add fixed, uniform, log-normal, or chunked delays to simulate latency
Proxying Forward unmatched requests to a target URL, enabling partial virtualization
Verification Assert that specific requests were made (count, order, timing)
Admin REST API Full API at /__admin for creating, updating, and deleting stubs at runtime
HTTPS Support Built-in TLS with self-signed certificates or custom keystores

2.3 Operating Modes

Standalone Server

Run WireMock as a self-contained JAR that serves virtual APIs. Ideal for shared team environments, Docker deployments, and CI/CD pipelines.

java -jar wiremock-standalone-3.x.x.jar --port 8080 --verbose

JUnit Rule / Extension (Embedded)

Embed WireMock into your Java test suite using JUnit 4 Rules or JUnit 5 Extensions. The server starts and stops automatically with each test class or method.

@WireMockTest(httpPort = 8080)
class MyApiTest {
    // WireMock is started automatically
}

Docker Container

Run WireMock as a Docker container with mappings mounted as volumes. Perfect for CI pipelines and team-shared environments.

docker run -d --name wiremock \
  -p 8080:8080 \
  -v $PWD/mappings:/home/wiremock/mappings \
  -v $PWD/__files:/home/wiremock/__files \
  wiremock/wiremock:3.9.2

3.3 Java / JUnit Integration

JUnit 5 Extension

import com.github.tomakehurst.wiremock.junit5.WireMockExtension;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.RegisterExtension;

import static com.github.tomakehurst.wiremock.client.WireMock.*;
import static com.github.tomakehurst.wiremock.core.WireMockConfiguration.wireMockConfig;

class UserServiceTest {

    @RegisterExtension
    static WireMockExtension wireMock = WireMockExtension.newInstance()
        .options(wireMockConfig().dynamicPort())
        .build();

    @Test
    void shouldFetchUserDetails() {
        // Arrange: configure stub
        wireMock.stubFor(get(urlPathEqualTo("/api/users/42"))
            .willReturn(aResponse()
                .withStatus(200)
                .withHeader("Content-Type", "application/json")
                .withBody("""
                    {
                        "id": 42,
                        "name": "Jane Doe",
                        "email": "jane@example.com"
                    }
                    """)));

        // Act: call the service under test
        String baseUrl = wireMock.baseUrl();
        UserService userService = new UserService(baseUrl);
        User user = userService.getUser(42);

        // Assert: verify response
        assertThat(user.getName()).isEqualTo("Jane Doe");

        // Verify: WireMock received the expected request
        wireMock.verify(1, getRequestedFor(urlPathEqualTo("/api/users/42")));
    }
}

4. Using WireMock in Testing

4.1 Stubbing Basics

Stubs define the expected request pattern and the response WireMock should return. Stubs can be configured via JSON files or the programmatic API.

JSON Stub Mapping

Place this file in the mappings/ directory:

{
    "id": "get-user-by-id",
    "name": "Get User by ID",
    "request": {
        "method": "GET",
        "urlPathPattern": "/api/users/[0-9]+"
    },
    "response": {
        "status": 200,
        "headers": {
            "Content-Type": "application/json"
        },
        "jsonBody": {
            "id": 1,
            "name": "John Smith",
            "email": "john@example.com",
            "role": "admin"
        }
    },
    "priority": 5
}

REST API (Runtime Configuration)

# Create a stub via the Admin API
curl -X POST http://localhost:8080/__admin/mappings \
  -H "Content-Type: application/json" \
  -d '{
    "request": {
        "method": "POST",
        "urlPath": "/api/orders",
        "headers": {
            "Content-Type": { "equalTo": "application/json" }
        }
    },
    "response": {
        "status": 201,
        "jsonBody": {
            "orderId": "ORD-12345",
            "status": "CREATED"
        }
    }
  }'

4.2 Request Matching

WireMock's matching engine supports a rich set of matchers for precise request identification:

URL Matching

// Exact URL match (including query string)
stubFor(get(urlEqualTo("/api/users?page=1&size=10"))
    .willReturn(ok()));

// URL path only (ignoring query params)
stubFor(get(urlPathEqualTo("/api/users"))
    .willReturn(ok()));

// Regex URL pattern
stubFor(get(urlPathMatching("/api/users/[0-9]+"))
    .willReturn(ok()));

// URL path with specific query parameter matching
stubFor(get(urlPathEqualTo("/api/search"))
    .withQueryParam("q", containing("wiremock"))
    .withQueryParam("limit", matching("[0-9]+"))
    .willReturn(ok()));

Header Matching

stubFor(get(urlPathEqualTo("/api/secure-data"))
    .withHeader("Authorization", matching("Bearer .+"))
    .withHeader("Accept", equalTo("application/json"))
    .withHeader("X-Request-Id", matching("[a-f0-9-]{36}"))
    .willReturn(ok()));

Request Body Matching

// JSON body matching with JSONPath
stubFor(post(urlPathEqualTo("/api/orders"))
    .withRequestBody(matchingJsonPath("$.customerId"))
    .withRequestBody(matchingJsonPath("$.items[?(@.quantity > 0)]"))
    .willReturn(created()));

// Exact JSON body match (ignoring key order and extra fields)
stubFor(post(urlPathEqualTo("/api/users"))
    .withRequestBody(equalToJson("""
        { "name": "Jane", "email": "jane@example.com" }
    """, true, true))
    .willReturn(created()));

// XML body matching with XPath
stubFor(post(urlPathEqualTo("/api/soap-endpoint"))
    .withRequestBody(matchingXPath("//OrderId[text()='12345']"))
    .willReturn(ok()));

JSON Stub: Complete Matching Example

{
    "request": {
        "method": "POST",
        "urlPath": "/api/payments",
        "headers": {
            "Content-Type": { "equalTo": "application/json" },
            "Authorization": { "matches": "Bearer .+" },
            "X-Idempotency-Key": { "matches": "[a-f0-9-]{36}" }
        },
        "bodyPatterns": [
            { "matchesJsonPath": "$.amount" },
            { "matchesJsonPath": "$.currency" },
            { "matchesJsonPath": "$.merchantId" }
        ]
    },
    "response": {
        "status": 200,
        "headers": {
            "Content-Type": "application/json"
        },
        "jsonBody": {
            "transactionId": "TXN-98765",
            "status": "APPROVED",
            "amount": 99.99
        }
    }
}

4.3 Response Templating

Response templating allows you to generate dynamic responses based on the incoming request. WireMock uses Handlebars as its templating engine.

{
    "request": {
        "method": "GET",
        "urlPathPattern": "/api/users/([0-9]+)"
    },
    "response": {
        "status": 200,
        "headers": {
            "Content-Type": "application/json"
        },
        "body": "{ \"id\": {{request.path.[2]}}, \"name\": \"User {{request.path.[2]}}\", \"requestedAt\": \"{{now}}\" }",
        "transformers": ["response-template"]
    }
}

Common Template Helpers

Helper Description Example
{{request.url}} Full request URL /api/users/42?include=profile
{{request.path.[n]}} URL path segment by index {{request.path.[2]}} → 42
{{request.query.param}} Query parameter value {{request.query.page}} → 1
{{request.headers.Name}} Request header value {{request.headers.X-Request-Id}}
{{request.body}} Raw request body Full body string
{{jsonPath request.body '$.field'}} Extract JSON field from body Dynamic field extraction
{{now}} Current timestamp 2026-04-01T10:30:00Z
{{randomValue type='UUID'}} Generate random UUID a1b2c3d4-e5f6-...
{{math 1 '+' 1}} Basic arithmetic 2

Advanced Templating Example: Echo Service

{
    "request": {
        "method": "POST",
        "urlPath": "/api/echo"
    },
    "response": {
        "status": 200,
        "headers": {
            "Content-Type": "application/json"
        },
        "body": "{\n  \"echo\": {\n    \"method\": \"{{request.method}}\",\n    \"url\": \"{{request.url}}\",\n    \"body\": {{jsonPath request.body '$'}},\n    \"timestamp\": \"{{now format='yyyy-MM-dd\\'T\\'HH:mm:ss.SSSZ'}}\",\n    \"correlationId\": \"{{randomValue type='UUID'}}\"\n  }\n}",
        "transformers": ["response-template"]
    }
}

4.4 Record & Playback

WireMock can act as a proxy, recording real API interactions and saving them as stub mappings for future playback. This is the fastest way to build a realistic set of virtual service responses.

Recording via CLI

# Start WireMock in recording mode, proxying to the real API
java -jar wiremock.jar \
  --port 8080 \
  --proxy-all="https://api.real-service.com" \
  --record-mappings \
  --verbose

All requests routed through WireMock will be forwarded to the real API. Both the request pattern and the response are saved as JSON in the mappings/ and __files/ directories.

Recording via REST API

# Start recording
curl -X POST http://localhost:8080/__admin/recordings/start \
  -H "Content-Type: application/json" \
  -d '{
    "targetBaseUrl": "https://api.real-service.com",
    "filters": {
        "urlPathPattern": "/api/.*"
    },
    "captureHeaders": {
        "Content-Type": {},
        "Accept": {}
    },
    "requestBodyPattern": {
        "matcher": "equalToJson",
        "ignoreArrayOrder": true,
        "ignoreExtraElements": true
    }
  }'

# ... run your tests or make API calls ...

# Stop recording and save mappings
curl -X POST http://localhost:8080/__admin/recordings/stop

Important: After recording, review the generated mappings to remove sensitive data (API keys, tokens, PII) before committing to source control. Use response templating to replace hardcoded values with dynamic placeholders.

4.5 Stateful Behavior (Scenarios)

WireMock scenarios allow you to model stateful interactions where the response changes based on the sequence of requests. This is essential for testing workflows like order processing, authentication flows, or multi-step wizards.

// Scenario: Order lifecycle
// State 1: Order is "PENDING" when first created
stubFor(get(urlPathEqualTo("/api/orders/100"))
    .inScenario("Order Lifecycle")
    .whenScenarioStateIs(Scenario.STARTED)
    .willReturn(aResponse()
        .withStatus(200)
        .withBody("{\"orderId\": 100, \"status\": \"PENDING\"}")));

// Transition: Confirming the order moves state to "CONFIRMED"
stubFor(post(urlPathEqualTo("/api/orders/100/confirm"))
    .inScenario("Order Lifecycle")
    .whenScenarioStateIs(Scenario.STARTED)
    .willSetStateTo("confirmed")
    .willReturn(aResponse().withStatus(200)));

// State 2: After confirmation, order shows as "CONFIRMED"
stubFor(get(urlPathEqualTo("/api/orders/100"))
    .inScenario("Order Lifecycle")
    .whenScenarioStateIs("confirmed")
    .willReturn(aResponse()
        .withStatus(200)
        .withBody("{\"orderId\": 100, \"status\": \"CONFIRMED\"}")));

// Transition: Shipping the order moves state to "SHIPPED"
stubFor(post(urlPathEqualTo("/api/orders/100/ship"))
    .inScenario("Order Lifecycle")
    .whenScenarioStateIs("confirmed")
    .willSetStateTo("shipped")
    .willReturn(aResponse().withStatus(200)));

// State 3: After shipping, order shows as "SHIPPED"
stubFor(get(urlPathEqualTo("/api/orders/100"))
    .inScenario("Order Lifecycle")
    .whenScenarioStateIs("shipped")
    .willReturn(aResponse()
        .withStatus(200)
        .withBody("{\"orderId\": 100, \"status\": \"SHIPPED\"}")));

4.6 Fault & Delay Simulation

Testing how your application handles failures is critical for resilience. WireMock allows you to simulate a wide range of network and service failures.

Delays

// Fixed delay (milliseconds)
stubFor(get(urlPathEqualTo("/api/slow-endpoint"))
    .willReturn(ok()
        .withFixedDelay(3000)));  // 3 second delay

// Random delay (uniform distribution)
stubFor(get(urlPathEqualTo("/api/variable-latency"))
    .willReturn(ok()
        .withUniformRandomDelay(500, 2000)));  // 500ms - 2000ms

// Log-normal delay (realistic latency distribution)
stubFor(get(urlPathEqualTo("/api/realistic-latency"))
    .willReturn(ok()
        .withLogNormalRandomDelay(90, 0.4)));  // median 90ms

// Chunked dribble delay (slow response body delivery)
stubFor(get(urlPathEqualTo("/api/slow-download"))
    .willReturn(ok()
        .withBody("large response body here...")
        .withChunkedDribbleDelay(5, 5000)));  // 5 chunks over 5 seconds

Faults

// Connection reset (TCP RST)
stubFor(get(urlPathEqualTo("/api/connection-reset"))
    .willReturn(aResponse().withFault(Fault.CONNECTION_RESET_BY_PEER)));

// Empty response (connection closed without data)
stubFor(get(urlPathEqualTo("/api/empty-response"))
    .willReturn(aResponse().withFault(Fault.EMPTY_RESPONSE)));

// Malformed response (random garbage bytes)
stubFor(get(urlPathEqualTo("/api/garbage"))
    .willReturn(aResponse().withFault(Fault.MALFORMED_RESPONSE_CHUNK)));

JSON Fault Configuration

{
    "request": {
        "method": "GET",
        "urlPath": "/api/timeout-simulation"
    },
    "response": {
        "status": 504,
        "fixedDelayMilliseconds": 30000,
        "headers": {
            "Content-Type": "application/json"
        },
        "jsonBody": {
            "error": "Gateway Timeout",
            "message": "Upstream service did not respond in time"
        }
    }
}

4.7 Request Verification

WireMock keeps a journal of all incoming requests, allowing you to verify that specific interactions occurred during a test.

Programmatic Verification (Java)

// Verify a request was made exactly once
verify(1, getRequestedFor(urlPathEqualTo("/api/users/42"))
    .withHeader("Accept", equalTo("application/json")));

// Verify a POST was made with specific body content
verify(postRequestedFor(urlPathEqualTo("/api/orders"))
    .withRequestBody(matchingJsonPath("$.customerId", equalTo("CUST-001")))
    .withRequestBody(matchingJsonPath("$.items.length()", equalTo("3"))));

// Verify no requests were made to a specific endpoint
verify(0, deleteRequestedFor(urlPathMatching("/api/users/.*")));

// Verify request timing
verify(getRequestedFor(urlPathEqualTo("/api/health"))
    .withHeader("X-Correlation-Id", matching(".+")));

// Get all requests for inspection
List<LoggedRequest> requests = findAll(
    getRequestedFor(urlPathMatching("/api/.*")));
assertThat(requests).hasSize(5);

REST API Verification

# Count requests to an endpoint
curl -X POST http://localhost:8080/__admin/requests/count \
  -H "Content-Type: application/json" \
  -d '{
    "method": "GET",
    "urlPathPattern": "/api/users/.*"
  }'

# Find all logged requests
curl http://localhost:8080/__admin/requests

# Find unmatched requests (useful for debugging)
curl http://localhost:8080/__admin/requests/unmatched

5. Testing Strategies with WireMock

5.1 Unit / Component Testing

Use WireMock to isolate components that make HTTP calls to external services. This gives you the speed of a unit test with the confidence of testing real HTTP interactions.

┌──────────────────────────────────────────────────────────────┐ │ Test Process (JVM) │ │ │ │ ┌──────────────┐ HTTP ┌──────────────────────────┐ │ │ │ Component │ ─────────▶ │ WireMock (embedded) │ │ │ │ Under Test │ ◀───────── │ Port: dynamic │ │ │ └──────────────┘ │ Lifecycle: per-test │ │ │ └──────────────────────────┘ │ └──────────────────────────────────────────────────────────────┘ 
@WireMockTest(httpPort = 0)  // dynamic port
class PaymentGatewayClientTest {

    @Test
    void shouldHandlePaymentApproval(WireMockRuntimeInfo wmInfo) {
        stubFor(post("/api/payments/charge")
            .willReturn(okJson("""
                {"transactionId": "TXN-001", "status": "APPROVED"}
            """)));

        PaymentClient client = new PaymentClient(wmInfo.getHttpBaseUrl());
        PaymentResult result = client.charge(new PaymentRequest("100.00", "USD"));

        assertThat(result.getStatus()).isEqualTo("APPROVED");
    }

    @Test
    void shouldHandlePaymentDecline(WireMockRuntimeInfo wmInfo) {
        stubFor(post("/api/payments/charge")
            .willReturn(jsonResponse("""
                {"transactionId": "TXN-002", "status": "DECLINED", "reason": "Insufficient funds"}
            """, 402)));

        PaymentClient client = new PaymentClient(wmInfo.getHttpBaseUrl());

        assertThatThrownBy(() -> client.charge(new PaymentRequest("100.00", "USD")))
            .isInstanceOf(PaymentDeclinedException.class)
            .hasMessageContaining("Insufficient funds");
    }

    @Test
    void shouldHandleServiceTimeout(WireMockRuntimeInfo wmInfo) {
        stubFor(post("/api/payments/charge")
            .willReturn(ok().withFixedDelay(10000)));  // 10s delay

        PaymentClient client = new PaymentClient(wmInfo.getHttpBaseUrl());
        client.setTimeoutMs(2000);  // 2s timeout

        assertThatThrownBy(() -> client.charge(new PaymentRequest("100.00", "USD")))
            .isInstanceOf(TimeoutException.class);
    }
}

5.2 Integration Testing

For integration tests, run WireMock as a standalone server alongside your application (or via Docker Compose). This tests the full application stack while controlling external dependencies.

Spring Boot Integration Test

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@WireMockTest(httpPort = 9090)
class OrderServiceIntegrationTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("payment.service.url", () -> "http://localhost:9090");
        registry.add("inventory.service.url", () -> "http://localhost:9090");
    }

    @Test
    void shouldCreateOrderSuccessfully() {
        // Stub the payment service
        stubFor(post(urlPathEqualTo("/api/payments/authorize"))
            .willReturn(okJson("{\"authorized\": true, \"authCode\": \"AUTH-999\"}")));

        // Stub the inventory service
        stubFor(post(urlPathEqualTo("/api/inventory/reserve"))
            .willReturn(okJson("{\"reserved\": true, \"reservationId\": \"RES-555\"}")));

        // Call the application's order endpoint
        OrderRequest orderRequest = new OrderRequest("CUST-001",
            List.of(new OrderItem("PROD-A", 2)));

        ResponseEntity<OrderResponse> response = restTemplate
            .postForEntity("/api/orders", orderRequest, OrderResponse.class);

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.CREATED);
        assertThat(response.getBody().getStatus()).isEqualTo("CONFIRMED");

        // Verify interactions with downstream services
        verify(1, postRequestedFor(urlPathEqualTo("/api/payments/authorize")));
        verify(1, postRequestedFor(urlPathEqualTo("/api/inventory/reserve")));
    }
}

5.4 Performance & Resilience Testing

WireMock enables controlled performance and resilience testing by simulating various failure modes and latency patterns:

Test Scenario WireMock Configuration What It Validates
High latency withFixedDelay(5000) Timeout handling, circuit breaker activation
Intermittent failures Scenario toggling between 200 and 503 Retry logic, exponential backoff
Connection drops withFault(CONNECTION_RESET_BY_PEER) Connection pool recovery
Rate limiting Return 429 with Retry-After header Rate limit handling, backoff strategy
Partial responses withChunkedDribbleDelay Streaming/chunked response handling
Service degradation Scenario: healthy → degraded → down Graceful degradation, fallback mechanisms

Circuit Breaker Testing Example

@Test
void shouldOpenCircuitBreakerAfterConsecutiveFailures() {
    // First 3 calls: service returns 503
    stubFor(get(urlPathEqualTo("/api/data"))
        .inScenario("circuit-breaker")
        .whenScenarioStateIs(Scenario.STARTED)
        .willReturn(serviceUnavailable())
        .willSetStateTo("failure-1"));

    stubFor(get(urlPathEqualTo("/api/data"))
        .inScenario("circuit-breaker")
        .whenScenarioStateIs("failure-1")
        .willReturn(serviceUnavailable())
        .willSetStateTo("failure-2"));

    stubFor(get(urlPathEqualTo("/api/data"))
        .inScenario("circuit-breaker")
        .whenScenarioStateIs("failure-2")
        .willReturn(serviceUnavailable())
        .willSetStateTo("failure-3"));

    // 4th call onwards: service is healthy again
    stubFor(get(urlPathEqualTo("/api/data"))
        .inScenario("circuit-breaker")
        .whenScenarioStateIs("failure-3")
        .willReturn(ok("{\"data\": \"recovered\"}")));

    DataClient client = new DataClient(wireMock.baseUrl());

    // Trigger failures
    assertThrows(ServiceException.class, () -> client.fetchData());
    assertThrows(ServiceException.class, () -> client.fetchData());
    assertThrows(ServiceException.class, () -> client.fetchData());

    // Circuit breaker should be OPEN - returns fallback without calling service
    DataResponse fallback = client.fetchData();
    assertThat(fallback.getSource()).isEqualTo("CACHE");

    // Verify WireMock only received 3 requests (circuit breaker prevented the 4th)
    verify(3, getRequestedFor(urlPathEqualTo("/api/data")));
}

6. CI/CD Integration with GitHub Actions

6.1 Pipeline Strategy

There are two primary approaches to integrating WireMock into a CI/CD pipeline:

Approach How Best For
Embedded (JUnit) WireMock starts/stops within the test JVM Java projects, unit/component tests, simplest setup
Standalone JAR Download JAR and start as a background process Multi-language projects, any stack with HTTP client tests
Docker Service Run as a GitHub Actions service container Docker Compose workflows, complex multi-service setups

6.2 Standalone WireMock Workflow

This workflow downloads WireMock, starts it as a background process, runs tests, and then stops it.

# .github/workflows/integration-tests.yml
name: Integration Tests with WireMock

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  WIREMOCK_VERSION: "3.9.2"
  WIREMOCK_PORT: 8080

jobs:
  integration-tests:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Java 21
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '21'
          cache: 'maven'

      # ---- WireMock Setup ----
      - name: Cache WireMock JAR
        uses: actions/cache@v4
        with:
          path: wiremock.jar
          key: wiremock-${{ env.WIREMOCK_VERSION }}

      - name: Download WireMock
        run: |
          if [ ! -f wiremock.jar ]; then
            curl -L -o wiremock.jar \
              "https://repo1.maven.org/maven2/org/wiremock/wiremock-standalone/${WIREMOCK_VERSION}/wiremock-standalone-${WIREMOCK_VERSION}.jar"
          fi

      - name: Start WireMock
        run: |
          java -jar wiremock.jar \
            --port ${{ env.WIREMOCK_PORT }} \
            --root-dir ./wiremock-config \
            --global-response-templating \
            --verbose &

          # Wait for WireMock to be ready
          echo "Waiting for WireMock to start..."
          for i in $(seq 1 30); do
            if curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/mappings > /dev/null 2>&1; then
              echo "WireMock is ready!"
              break
            fi
            echo "Attempt $i/30 - waiting..."
            sleep 1
          done

      - name: Verify WireMock is running
        run: |
          curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/mappings | head -20
          echo ""
          echo "Loaded mappings count:"
          curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/mappings/count

      # ---- Run Tests ----
      - name: Run integration tests
        run: mvn verify -Dtest.wiremock.url=http://localhost:${{ env.WIREMOCK_PORT }}
        env:
          EXTERNAL_API_URL: http://localhost:${{ env.WIREMOCK_PORT }}

      # ---- Diagnostics (on failure) ----
      - name: Dump WireMock logs on failure
        if: failure()
        run: |
          echo "=== Unmatched Requests ==="
          curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/requests/unmatched
          echo ""
          echo "=== All Requests ==="
          curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/requests
          echo ""
          echo "=== Near Misses ==="
          curl -s http://localhost:${{ env.WIREMOCK_PORT }}/__admin/near-misses/request

6.3 Docker-Based Workflow

Use GitHub Actions service containers to run WireMock as a Docker container alongside your tests.

# .github/workflows/docker-integration-tests.yml
name: Integration Tests (Docker WireMock)

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  integration-tests:
    runs-on: ubuntu-latest

    services:
      wiremock:
        image: wiremock/wiremock:3.9.2
        ports:
          - 8080:8080
        options: >-
          --health-cmd "curl -f http://localhost:8080/__admin/mappings || exit 1"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
        volumes:
          - ${{ github.workspace }}/wiremock-config/mappings:/home/wiremock/mappings
          - ${{ github.workspace }}/wiremock-config/__files:/home/wiremock/__files

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Java 21
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '21'
          cache: 'maven'

      - name: Wait for WireMock service
        run: |
          echo "Waiting for WireMock service..."
          for i in $(seq 1 30); do
            if curl -s http://localhost:8080/__admin/mappings > /dev/null 2>&1; then
              echo "WireMock is ready!"
              break
            fi
            sleep 2
          done

      - name: Load additional mappings (optional runtime stubs)
        run: |
          # Load any environment-specific stubs via the REST API
          for file in wiremock-config/ci-mappings/*.json; do
            if [ -f "$file" ]; then
              echo "Loading mapping: $file"
              curl -s -X POST http://localhost:8080/__admin/mappings \
                -H "Content-Type: application/json" \
                -d @"$file"
            fi
          done

      - name: Run integration tests
        run: mvn verify -Pintegration-tests
        env:
          PAYMENT_SERVICE_URL: http://localhost:8080
          INVENTORY_SERVICE_URL: http://localhost:8080

      - name: Collect WireMock diagnostics
        if: failure()
        run: |
          echo "=== Unmatched Requests ==="
          curl -s http://localhost:8080/__admin/requests/unmatched | jq '.'
          echo "=== Near Misses ==="
          curl -s http://localhost:8080/__admin/near-misses/request | jq '.'

Note: GitHub Actions service containers run in the same Docker network as the runner. Access them via localhost on the mapped port. Volume mounts require the files to exist in the workspace before the service starts, so the checkout step must come first.

7. Best Practices & Anti-Patterns

Best Practices

  • Version your mappings: Store all WireMock configurations in source control alongside your tests
  • Organize by service: Use separate directories per external service for clarity and ownership
  • Use response templating: Avoid hardcoding dynamic values; use Handlebars templates for dates, IDs, and request-derived data
  • Set priorities: Use the priority field to control match order, especially when using regex patterns (lower number = higher priority)
  • Verify interactions: Always verify that expected requests were made — don't just test the happy path response
  • Test failure modes: Use fault injection and delays to validate resilience patterns (retries, circuit breakers, timeouts)
  • Use dynamic ports in tests: Avoid port conflicts in CI by using dynamicPort() for embedded WireMock
  • Clean up between tests: Use @BeforeEach or WireMock.reset() to ensure test isolation
  • Record from real APIs periodically: Keep your virtual services in sync with actual API behavior
  • Add diagnostics on failure: Log unmatched requests and near misses in CI to accelerate debugging

Anti-Patterns to Avoid

Common mistakes that reduce the value of service virtualization:

Anti-Pattern Why It's Harmful Better Approach
Over-matching Overly specific stubs break when irrelevant request details change Match on the minimum necessary criteria (URL path, key headers, essential body fields)
Stale mappings Virtual service drifts from real API, giving false confidence Periodically re-record from the real API; add contract tests
Hardcoded IDs/dates Tests become brittle and time-sensitive Use response templating with {{randomValue}} and {{now}}
Single global WireMock Shared state causes test interference and flakiness Use per-test or per-class WireMock instances with dynamic ports
No verification Tests pass even when the application doesn't call the API Always verify expected requests were made with correct parameters
Testing only happy paths Misses the primary value of service virtualization: controlled failure testing Create stubs for 4xx, 5xx, timeouts, malformed responses, and connection failures
Secrets in mappings API keys and tokens leaked in source control Use pattern matching for auth headers (matches: "Bearer .+") instead of exact values
